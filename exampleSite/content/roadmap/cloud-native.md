如果只记一条主线，按这个顺序由下往上爬：

| 层级 | 技术 | 关注点 |
| --- | --- | --- |
| 1 | Linux | 系统、网络、排障 |
| 2 | 容器 | Docker / containerd、镜像、运行时 |
| 3 | Kubernetes | 编排、调度、声明式 API |
| 4 | GitOps | Argo CD / Flux，持续交付 |
| 5 | 可观测 | Metrics / Logs / Traces 三支柱 |
| 6 | 服务网格 | 流量治理、零信任 |
| 7 | 平台工程 | IDP、黄金路径、自服务 |

每往上一层，前提是下一层你能调得动、查得清。跳级学很爽，但出事时会原形毕露。

## 怎么用这条线

- **自检**：对照每一层，诚实问自己"出事时我能不能独立定位"。能，再往上。
- **不跳级**：K8s 出问题十有八九是下面的网络 / 存储 / Linux 问题，跳过基础迟早补课。
- **结合项目**：每爬一层，找一个真实场景练手，别停在看文档。

> 工具会过时，底层原理不会。把每一层的"为什么"搞明白，换工具只是换语法。
