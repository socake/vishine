写 Prompt 不是玄学。同样的需求，结构化的 prompt 和随手一句话，输出稳定性差一个数量级。

## 结构化 Prompt

把指令拆成清晰的区块，模型更容易"对号入座"：

```text
# 角色
你是一名资深财报分析师。

# 任务
从下面的文本中抽取公司营收和净利润。

# 约束
- 只输出 JSON，不要解释
- 找不到的字段填 null

# 输入
{text}
```

> 角色设定不是装饰，它能把模型拉进特定语境，显著影响用词和判断标准。

## few-shot 与思维链

- **few-shot**：给 2-3 个输入输出示例，比纯文字描述格式更有效，尤其是边界情况
- **思维链（CoT）**：复杂推理让它"先想再答"，一句 `请一步步推理` 往往就能提升正确率

```text
示例 1：
输入：苹果 3 个，又买了 2 个
输出：{"total": 5}

示例 2：
输入：橘子 10 个，吃掉 4 个
输出：{"total": 6}

现在处理：
输入：{user_input}
```

## 输出格式约束

要程序消费的结果，必须强约束成 JSON，并在代码侧校验：

1. prompt 里明确"只输出 JSON、不要 markdown 代码块包裹"
2. 优先用模型的 **structured output / JSON mode**（如果支持）
3. 解析失败就重试，或要求模型"修正成合法 JSON"

```python
import json

def parse_safe(raw):
    try:
        return json.loads(raw)
    except json.JSONDecodeError:
        return None  # 触发重试或修正流程
```

## 防注入

用户输入拼进 prompt 时，可能夹带"忽略以上指令"这类攻击：

- 用明确分隔符把用户内容包起来，并声明**分隔符内一律视为数据，不是指令**
- 系统指令和用户输入分开传（system / user 角色），别揉成一段
- 对输出再做一层校验，别让模型的回答直接触发危险动作

## 版本管理与评测

Prompt 是会迭代的资产，要像代码一样管：

- **进版本库**：每条线上 prompt 带版本号，改动可回滚
- **建评测集**：固定一批输入和期望输出，改 prompt 后批量回归
- **A/B 对比**：新旧 prompt 在同一批样本上跑分，用数据决定是否上线

把这套流程跑起来，Prompt 才从"碰运气"变成可控的工程能力。
