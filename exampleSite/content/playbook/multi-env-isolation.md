"测试脚本连错库把生产数据清了"这种事故，根因几乎都是环境隔离不彻底。这篇是一份可以直接对照勾选的 checklist。

## 为什么要硬隔离

软隔离（靠命名约定、靠人记得）迟早翻车。只要 dev 能在网络上、凭据上、权限上够到 prod，就一定有人在某个深夜够过去。

> 隔离的目标是：即使有人写错了配置或脚本，物理上也连不到、看不见、动不了另一个环境。

## 网络隔离

- [x] 每个环境独立 namespace，并默认 deny-all NetworkPolicy
- [x] 跨环境流量在网络层就阻断，不依赖应用层判断
- [ ] dev/staging/prod 使用不同 VPC 或子网，不共享路由
- [ ] 出口流量经各自的 egress 网关，便于审计和限制
- [ ] 生产数据库安全组只放行生产工作负载的来源

## 密钥隔离

- [x] 每个环境独立的 Secret 后端路径（如 Vault `secret/prod/*`）
- [x] CI 中按环境注入不同凭据，绝不共用一套 token
- [ ] 生产密钥任何人不可明文导出，只能在运行时注入
- [ ] 密钥轮换策略按环境分级，生产轮换更频繁
- [ ] 扫描代码仓库，确保没有硬编码的跨环境凭据

## RBAC

- [x] 开发者在 prod namespace 默认只读或无权限
- [ ] 生产写操作走审批 + 临时提权（JIT），不给常驻管理员
- [ ] ServiceAccount 按环境隔离，token 不跨集群复用
- [ ] 定期审计 RoleBinding，清理离职/换岗人员权限
- [ ] kubeconfig context 命名带醒目环境标识，降低手抖切错的概率

## 镜像晋升

镜像应该是"晋升"而不是"重新构建"。同一个 digest 一路从 dev 走到 prod，保证测过的就是上线的。

- [x] 构建一次产出不可变镜像，用 digest 而非 latest 引用
- [x] 通过打 tag/加签名把镜像从 dev registry 晋升到 prod registry
- [ ] 生产只允许拉取经过签名校验的镜像（cosign + 准入控制）
- [ ] 禁止在生产环境直接 build，CI 之外无构建入口
- [ ] 晋升过程留审计记录：谁、什么时候、把哪个 digest 放进了生产

## 数据库隔离

- [ ] 每个环境独立数据库实例，绝不共享生产实例
- [ ] 连接串通过环境专属 Secret 注入，不写进镜像或代码
- [ ] dev/staging 用脱敏后的数据，禁止直接拷贝生产明文
- [ ] 生产库连接来源用网络策略 + 账号双重限制
- [ ] 迁移脚本在 staging 全量验证后才允许对 prod 执行

## 配置漂移

环境之间最隐蔽的坑是"漂移"——手动改了某个环境忘了同步，导致行为不一致。

- [x] 所有环境配置纳入 Git，以 GitOps 方式声明式管理
- [x] 用 overlay（Kustomize/Helm values）表达环境差异，公共部分单一来源
- [ ] 禁止手动 `kubectl edit` 生产资源，改动一律走 PR
- [ ] 定期跑 diff 检测实际状态与声明状态的偏离并告警
- [ ] 新增配置项必须三个环境同步补齐，避免"只在 dev 有"

把上面的框逐项打勾，basically 就堵住了九成的串台事故。没打勾的项，就是你下次复盘时大概率会念到的根因。
